## 核心论点

Harness 不是简单的包装，而是一套让自主智能体实现**自主思考、自主行动、自主修复**的完整系统，也是玩具级 Demo 与生产级智能体之间的本质区别。

综合 Anthropic、OpenAI、LangChain 及全球 AI 工程社区的最佳实践，生产级 Agent Harness 需要 **12 个完全独立、环环相扣的核心模块**。

---

## 为什么需要 Agent Harness？

- **行业痛点**：智能体演示时流畅无比，一到生产环境就原形毕露，任务成功率断崖式下跌
- **真相**：问题从来不在模型本身，而在模型周围的那套基础设施
- **LangChain 实验**：完全没改动模型权重和底层算法，只优化了 Harness 架构 → TerminalBench 2.0 从 30 名开外飙升至第 5 名
- **自主优化**：让 LLM 自主优化 Harness 架构，任务通过率冲到 76.4%，吊打所有人工设计的系统

### 定义

> Agent Harness 就是包裹大语言模型的一整套**操作系统级软件基础设施**。

LangChain 的 Vivek Trivedy 经典名言：**"If you're not the model, you're the harness."**

### 类比：计算机架构

| 组件 | Agent 对应 | 作用 |
|------|-----------|------|
| CPU | 裸 LLM | 核心计算能力，无法独立完成任务 |
| 临时内存 | 上下文窗口 | 速度快但容量有限 |
| 硬盘 | 向量数据库/长期存储 | 容量大但响应较慢 |
| 设备驱动 | 工具集成 | 让模型调用外部能力 |
| **操作系统** | **Agent Harness** | 让一切协同工作 |

Beren Millidge（2023）直言：我们通过 Agent Harness **重新发明了冯·诺依曼架构**。

## 三个工程层级（易混淆）

1. **提示工程**（Prompt Engineering）— 打磨模型接收的指令
2. **上下文工程**（Context Engineering）— 管理模型在不同阶段能看到哪些信息
3. **Harness 工程**（Harness Engineering）— 涵盖前两者 + 工具编排、状态持久化、错误恢复、验证循环、安全管控、生命周期管理等全套应用基础设施

## 十二大核心模块

### 1️⃣ 编排循环（Orchestration Loop）

整个智能体的**心跳**，所有行为的核心引擎。ReAct 循环、TAO（思考-行动-观察）循环都是其具体实现。

**运行逻辑**：
1. 组装完整提示词（系统指令 + 工具信息 + 记忆 + 对话历史）
2. 发送给大模型
3. 解析输出，判断是否需要调用工具
4. 执行工具调用，结果返回模型
5. 重复直到任务完成或触发终止条件

**Anthropic 的设计哲学**：称为**"笨循环"（dumb loop）** — 所有智能决策由模型完成，Harness 只负责按流程转场、调度任务，不参与核心推理。分工明确，降低系统复杂度。

---

### 2️⃣ 工具（Tools）

智能体的**手**，与现实世界交互的唯一途径。以标准化 Schema 形式定义（名称、描述、参数类型、返回格式）。

**工具层职责**：工具注册 → Schema 校验 → 参数提取 → 沙箱执行 → 结果捕获 → 格式化回传

**行业实践**：
- **Claude Code**：六大类工具（文件操作、搜索、命令执行、网页访问、代码智能、子智能体孵化）
- **OpenAI Agents SDK**：三类工具（函数调用、官方托管工具、MCP 服务器工具）

> 工具层的设计直接决定了智能体的**能力边界**。

---

### 3️⃣ 记忆（Memory）

智能体跨越时间尺度、保持任务连续性的关键。分为：

| 类型 | 说明 | 实现 |
|------|------|------|
| **短期记忆** | 单次会话对话历史 | 对话上下文 |
| **长期记忆** | 跨会话持久化 | Anthropic: claude.md + MEMORY.md；LangGraph: 按命名空间 JSON；OpenAI: SQLite/Redis |

**Claude Code 三级记忆层级**（行业标杆）：
1. **轻量级索引** — ~150 字符/条，常驻内存，快速响应
2. **详细主题文件** — 按需加载，平衡容量与速度
3. **原始交互记录** — 仅通过搜索访问，保证数据完整性

**核心原则**：智能体不会完全依赖记忆，行动前会与实际状态核对验证。

---

### 4️⃣ 上下文管理（Context Management）

生产级智能体最容易翻车的**重灾区**。

**核心问题 — 上下文腐烂（Context Rot）**：关键信息落在上下文窗口中间位置时，模型性能暴跌 30%+（斯坦福 "Lost in the Middle" + Chroma 实验）。

**四种应对策略**：

| 策略 | 做法 |
|------|------|
| **压缩**（Compaction） | 接近上限时摘要处理，保留核心决策，丢弃冗余输出 |
| **观察屏蔽**（Observation Masking） | 隐藏旧工具输出细节，保留调用记录 |
| **即时检索**（Just-in-time Retrieval） | 维护轻量级索引，动态加载（Claude Code 用 grep/glob/head/tail） |
| **子智能体委派** | 拆分复杂任务，只返回 1000-2000 Token 精简摘要 |

**终极目标**：找到**最小的高信噪比 Token 集合**。

---

### 5️⃣ 提示词组装（Prompt Assembly）

定义模型在每一轮推理中看到的**世界**。不是简单拼接，而是分层堆叠的结构化过程。

**标准组装顺序**（优先级从高到低）：
1. 系统提示词 — 身份和核心规则
2. 工具定义 — 可用能力
3. 记忆文件 — 历史经验
4. 对话历史 — 当前任务进度
5. 用户消息 — 最新需求

**OpenAI Codex 的优先级栈**：服务器系统消息 > 工具定义 > 开发者指令 > 用户指令 > 对话历史。

---

### 6️⃣ 工具调用与结构化输出（Tool Calling & Structured Output）

模型与 Harness 之间的**通用语言**。

- 模型直接返回标准化 `tool_calls` 结构化对象（工具名称 + 参数值）
- Harness 只需判断：有工具调用 → 执行并继续；无工具调用 → 终止循环
- OpenAI/LangChain 通过 Pydantic 框架做 Schema 约束
- 遗留方案：`RetryWithErrorOutputParser`（将原始提示 + 失败输出 + 错误返回模型自主修正，效率较低）

---

### 7️⃣ 状态与检查点（State & Checkpointing）

智能体实现**断点续跑、可回溯、可调试**的核心。

**残酷的数学事实**：10 步任务流程，每步 99% 成功率 → 端到端仅 90.4%。

**框架方案**：
- **LangGraph**：状态建模为类型化字典，归约器合并更新，检查点在超级步骤边界触发
- **OpenAI**：四种状态策略（应用内存、SDK 会话、服务器端对话 API、previous_response_id 链式调用）
- **Claude Code**：Git 提交作为检查点 + 进度文件作为结构化草稿本

---

### 8️⃣ 错误处理（Error Handling）

智能体在复杂环境中稳定运行的**安全网**。

**LangGraph 四类错误**：
1. **瞬时错误**（网络波动、API 限流）→ 带退避的重试
2. **模型可恢复错误**（参数错误、逻辑失误）→ 包装成工具消息返回模型
3. **用户可修复错误**（权限不足、配置错误）→ 中断等待人工输入
4. **意外错误**（系统崩溃、底层异常）→ 直接抛出便于调试

**Anthropic 策略**：工具处理器内部捕获所有失败，返回模型，确保主编排循环不中断。
**Stripe 策略**：重试限制在 2 次以内，避免资源耗尽。

---

### 9️⃣ 护栏（Guardrails）

智能体的**安全红线**，防止越权、有害、违规操作。

**OpenAI 三层防护**：
1. **输入护栏** — 过滤恶意和违规输入
2. **输出护栏** — 确保输出内容合规安全
3. **工具护栏** — 管控工具调用权限

触发绊线机制 → 智能体立即终止当前操作（紧急制动）。

**Anthropic 的彻底解耦**：权限执行与模型推理完全分离。模型只负责"想做什么"，工具系统负责"能做什么"。Claude Code 独立管控 ~40 种工具能力，三阶段把关（项目加载建立信任 → 每次调用前检查权限 → 高风险操作需用户确认）。

---

### 🔟 验证与反馈（Verification & Feedback）

**玩具级 vs 生产级的分水岭**。

**三种验证方式**：
1. **基于规则的反馈** — 测试用例、Linter、类型检查器
2. **视觉反馈** — Playwright 截图检查 UI 效果
3. **模型当裁判** — 独立子智能体评估主智能体输出

> Claude Code 创始人 Boris Cherny：给智能体加入验证机制，输出质量提升 **2-3 倍**。

验证循环不是额外开销，而是保证智能体产出价值的**必要投入**。

---

### 1️⃣1️⃣ 子 Agent 编排（Subagent Orchestration）

让单个智能体升级为**智能体集群**，解决复杂、大规模、多领域任务。

**Claude Code 三种模式**：
- **Fork** — 创建父上下文精确副本
- **Teammate** — 通过独立终端面板通信
- **Worktree** — 每个 Agent 分配独立 Git 工作树

**OpenAI SDK**：
- **Agents-as-tools** — 专家 Agent 处理细分任务
- **Handoffs** — 任务全面交接

**LangGraph**：子 Agent 实现为嵌套状态图，图结构管理任务流转。

---

### 1️⃣2️⃣ 初始化与环境搭建（Initialization & Environment Setup）

所有模块协同工作的**起点**。

**标准执行周期**：
1. 提示词组装 → 2. 模型推理 → 3. 输出分类 → 4. 工具执行（校验参数、检查权限、沙箱运行）→ 5. 结果打包 → 6. 上下文更新（触发压缩）→ 7. 回到第 1 步

**终止条件**：模型无工具调用 | 达到最大轮次 | Token 预算耗尽 | 护栏触发 | 用户中断 | 安全拒绝

## 五大主流框架对比

| 框架 | 设计哲学 | 编排方式 | 状态管理 | 多智能体 | 适合场景 |
|------|---------|---------|---------|---------|---------|
| **Anthropic Claude Agent SDK** | 薄 Harness、信任模型 | `query()` 笨循环，Gather-Act-Verify | Git 提交 | Fork/Teammate/Worktree | 轻量高效、与模型深度耦合 |
| **OpenAI Agents SDK** | 代码优先 | Runner 类，异步/同步/流式 | 四种策略 | Agents-as-tools/Handoffs | 快速开发、开发者友好 |
| **LangGraph** | 图结构 | 显式状态图 + 条件边 | 类型化字典 + 归约器 | 嵌套状态图 | 复杂多分支长流程 |
| **CrewAI** | 角色驱动 | 智能体/任务/团队解耦 + Flows | — | 角色分工 | 团队式多角色协作 |
| **AutoGen**（微软） | 对话驱动 | 顺序/并发/群组聊天/交接/magentic | — | 对话协议 | 开放式多智能体交互 |

> 核心模式高度趋同，但设计哲学截然不同。**没有绝对优劣，只有场景适配**。

## 脚手架隐喻与共同进化

Agent Harness 像建筑脚手架 — 模型能力越强，Harness 复杂度应越低。

**Manus 项目**：半年内重构 5 次，每次做减法（复杂工具定义 → 通用 Shell；管理智能体 → 结构化交接），性能反而持续提升。

**共同进化原则**：现代大模型后训练阶段会将特定 Harness 纳入训练循环，模型内化的能力越多，Harness 需要的封装就越少。

**面向未来的测试**：模型升级后，智能体性能应自然提升，无需增加 Harness 复杂度。

## 七大架构抉择

| # | 抉择 | 建议 |
|---|------|------|
| 1 | **单智能体 vs 多智能体** | 先榨干单智能体，工具重叠超 10 个或任务域明显分离时才拆分 |
| 2 | **ReAct vs 计划-执行** | ReAct 灵活但每步成本高；计划-执行分离规划与执行（LLMCompiler 快 3.6 倍） |
| 3 | **上下文管理策略** | 五种方法选组合，核心是保留推理痕迹、减少 Token 消耗 |
| 4 | **验证循环设计** | 计算式验证（测试/Linter）+ 推理验证（模型当裁判），两者结合最优 |
| 5 | **权限与安全** | 宽松模式高效但有风险，严格模式安全但低效，按部署场景平衡 |
| 6 | **工具范围** | 工具越多性能越差！Vercel 砍掉 80% 工具后性能反而提升，只暴露当前步骤所需最小工具集 |
| 7 | **Harness 厚度** | 薄 Harness 信任模型，厚 Harness 编码控制逻辑；模型越强越偏向薄 Harness |
